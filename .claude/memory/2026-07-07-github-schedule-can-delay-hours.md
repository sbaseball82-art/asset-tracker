GitHub Actionsのcronは数時間〜半日遅れることがある（実測11〜14時間）→「朝に画像がない＝壊れた」ではない。時刻が重要なワークフローはcronを2段構えにする。

## 状況

daily-stock-report（cron '30 22 * * *' = 7:30 JST）の実測実行時刻:
- 7/4 22:30 UTC 予定分 → 実際は 7/5 09:30 UTC（11時間遅れ）に実行され**成功**
- 7/5 22:30 UTC 予定分 → 実際は 7/6 12:59 UTC（14.5時間遅れ）に実行され**成功**

オーナーには「朝7:30に画像がない＝自動更新されていない」と見えるが、
ジョブ自体は遅れて成功していた。GitHub公式にも「高負荷時にスケジュールは
遅延・スキップされることがある」と明記されている（無料ランナーの仕様）。

## 誤った対処（やりがちなこと）

- 「壊れている」と判断してコードやワークフローの実行ロジックを修理し始める
  （まず Actions タブで**遅れて成功していないか**を確認する）
- 古い失敗runをRe-runする（別の教訓ファイル参照。Re-runは古いSHAで動くので別の失敗を生む）

## 正しい対処

1. 診断: Actionsタブ（または `mcp github actions_list`）で当該ワークフローの
   run一覧の `created_at` と conclusion を見る。「遅れて success」なら故障ではない
2. 緩和策: 締切がある生成物はcronを2本にする（2026-07-07 に適用済み）:
   ```yaml
   - cron: '15 21 * * *'   # 06:15 JST 本命
   - cron: '30 22 * * *'   # 07:30 JST 予備
   ```
   両方動いた日は2回コミットされるだけで無害（latest.png が新しい方で上書き）
3. それでも間に合わない朝は workflow_dispatch（Run workflowボタン）で即時実行できる
   （コミット工程が堅牢化済みなので、いつ手動実行しても安全）
